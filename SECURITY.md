# Security Policy

## Reporting a vulnerability

Please do not open public issues for security problems. Report privately via [GitHub Security Advisories](https://github.com/vynazevedo/sonarqube-selfhosted/security/advisories/new). You should receive a response within 7 days.

## Scope

This repository provisions infrastructure. Vulnerabilities in SonarQube itself should be reported to SonarSource. Relevant reports here include: privilege escalation through the IAM role, secret exposure, network exposure beyond ports 80/443, and unsafe defaults.

## Security model

- The instance is reachable only on ports 80 and 443 through Caddy. SonarQube and PostgreSQL are never published directly.
- There is no SSH access. Administration happens through SSM Session Manager.
- IMDSv2 is required and all EBS volumes are encrypted.
- The database password is generated by Terraform, stored as an SSM SecureString parameter and read by the instance at boot. It is never written to user data or tfvars.
- The IAM instance role can read exactly one SSM parameter and write to the backup bucket, nothing else beyond SSM management.

## Known trade-offs

- The generated database password is stored in the Terraform state, as with any Terraform-managed secret. Use an encrypted remote state backend with restricted access.
- The default `allowed_cidrs` is `0.0.0.0/0` because most teams need CI systems and developers to reach the server. Restrict it if your network layout allows.
